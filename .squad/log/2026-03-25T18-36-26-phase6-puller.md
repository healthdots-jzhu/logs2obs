## Overview

Building the pull-based log ingestion layer. Logs2Obs receives logs via HTTP push (Phase 2) and Kafka consumers (Phase 3); Phase 6 adds active pull capabilities from external sources (S3, Azure Blob, CloudWatch, HTTP endpoints).

**Value:** Integrate logs from systems that don't push to Logs2Obs. Enable backfill/replay scenarios. Time-shifted log collection.

## Technical Architecture

### Core Abstraction

```
IPullConnector
├── AwsS3Connector (S3 bucket polling)
├── AzureBlobConnector (Azure Blob Storage polling)
├── CloudWatchConnector (CloudWatch Logs API polling)
└── HttpConnector (Generic HTTP endpoint polling)
```

### Scheduler

- **Quartz.NET** periodic scheduling
- PullJobQuartzJob execution adapter
- PullJobStateService: Track cursors, prevent duplication
- Backfill support: Reset cursor to arbitrary point

### Integration Point

Pull jobs trigger IngestLogsHandler (existing from Phase 5), which dispatches to:
- StorageWriterWorker → MinIO Parquet writes
- SearchIndexerWorker → Meilisearch indexing

### State Persistence

- Cursor-based position tracking (S3 key, blob name, CloudWatch token, HTTP timestamp)
- Atomic state updates to prevent duplicate pulls
- Support for backfill (manual cursor reset) and replay

## Metrics & Observability

**OpenTelemetry Meters** (logs2obs.* prefix):
- `pull.frequency` — pulls per connector per interval
- `pull.connector.latency` — time to enumerate source
- `pull.items.processed` — total items pulled by connector
- `pull.items.failed` — retriable/non-retriable failures
