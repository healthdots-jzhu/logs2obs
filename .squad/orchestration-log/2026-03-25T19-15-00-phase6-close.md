## What Was Built

### Core Components

**Logs2Obs.Puller** (src/Logs2Obs.Puller/)

1. **IPullConnector Abstraction**
   - `AwsS3PullConnector` — Enumerates S3 bucket prefixes, parses NDJSON, tracks cursor via `lastProcessedKey`
   - `AzureBlobPullConnector` — Lists blobs under container prefix, parses NDJSON, tracks via `lastBlobName`
   - `CloudWatchPullConnector` — Polls CloudWatch Logs API (JSON array response), tracks via `lastEventTimestamp`
   - `HttpPullConnector` — Polls generic HTTP endpoint with OAuth2/API key auth, tracks via `lastPullTimestamp`

2. **Scheduler Integration**
   - `PullJobScheduler` — Quartz-based scheduler using cron expressions from `PullJobConfig.Schedule`
   - `PullJobQuartzJob` — Quartz job adapter that executes connector + state service sequentially
   - Startup loads enabled jobs and registers `CronTrigger` for each

3. **State Service**
   - `PullJobStateService` — Persists `PullJobConfig` (by JobId) and `PullJobState` (cursor data) via `IMetadataStore`
   - Atomic state updates prevent duplicate pulls
   - Supports backfill (manual cursor reset) and replay scenarios

4. **Observability**
   - `PullerMetrics` — OpenTelemetry meters (logs2obs.* prefix):
     - `pull.frequency` — pulls per connector per interval
     - `pull.connector.latency` — time to enumerate source
     - `pull.items.processed` — total items pulled by connector
     - `pull.items.failed` — retriable/non-retriable failures

5. **Program.cs**
   - Standalone pull service with DI registration
   - Loads `PullerOptions` from config
   - Registers all 4 connectors, scheduler, state service, and metrics

## Key Technical Decisions

1. **State Persistence Keys** — Introduced `PullStateRecord` and `PullJobStateRecord` wrappers for `IMetadataStore` key extraction convention. Keys: `pullstate:{jobId}`, `pulljob:{tenantId}:{jobId}`.

2. **Quartz Concurrency** — Bound job execution via `PullerOptions.MaxConcurrentJobs` + Quartz thread pool configuration. Single config point for max concurrent pulls.

3. **Batch Model Reuse** — Puller publishes `Logs2Obs.Worker.Models.LogEntryBatch` to avoid duplication. Batch model already exists and is battle-tested (Phase 5).

4. **Queue Configuration** — Publish target (`StorageWriterQueue`) read from `PullerOptions` instead of hardcoded. Enables local/dev/prod switching.

5. **Connector Input Parsing** — 
   - S3/Azure: Treat bucket/container as prefixes
   - CloudWatch: Expects JSON array of `LogEntry` items
   - Http/S3/Azure: NDJSON parsing with tenant header override
